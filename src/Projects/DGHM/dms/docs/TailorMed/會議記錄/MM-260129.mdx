## 三、討論重點摘要

- 1. Quotation 在資料庫裡開立不太方便，它會自己跑上跑下
  - A:因為使用 Sort 功能緣故，開立之一筆預設為 TM111111，在自訂單號(#)前它會因為數字較小排序在上。
- 2. 操作人帳號命名更改
  - A:使用 Database 左下角頭象功能，進到 Account overview 裡的 Edit name 進行修改。
    ![PIC](./0129-01.png)
    ![PIC](./0129-02.jpg)
  - ⭐ 建議一人一帳號，避免權限混淆，也可以追溯操作人責任。
- 3. Sale 帳號有限度編輯功能
  - A:Sales Rep. 身份若要使用 Interface 進行 Quotation 中 Status 狀態進行業務狀態回報，就需增設 Editor 權限，此部份會有相關軟體租賃費用的產生，詳細報價後續一併提出。
    ![PIC](./0129-03.jpg)

- 4. OMS 的填單量相較於 CRM 及 FIN 操作來說多，因此資料庫為公司工作的核心，相應的文件也有齊備之必要。
  - A: 建議若需要更有效率完成填單工作，可以考慮使用 Automation 機制來進行，以減少人工操作的時間。同時再輔以 Interface 操作介面，讓使用者可以在同一畫面進行各式填單工作的一次性完成。減少不同資料表的切換。

- 5. 歷史資料登錄
  - A: 考量到當前資料庫的應用，期望能夠將近一年之訂單資料登錄進入資料庫，以供後續業務人員進行業務分析及歷史業績檢視，同時也可以作為公司發展的相關數據參考。
  - 初步分析在過往歷史資料訂單中，要進行內容回填部份有包含：1.Quotation 2.Order, 3.AIRWAYBILL, 4.Debit Note. 在不計入 Release Check List for Collection 與 Sample R/R Checklist ...等的情況下，總共要完成的數量為 1,200 (TM-111400 ~ TM111699) 筆。
  - 其他需注意事項：歷史資料實際的內容以數位檔案為主，但其使用的語言別以及格式均沒有統一，因此資料的整理可能的複雜度較高會再提升。
  - 後續評估作業的複雜度及時間，再行報價。
