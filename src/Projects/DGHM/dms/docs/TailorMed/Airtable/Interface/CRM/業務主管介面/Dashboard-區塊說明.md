# Dashboard 區塊說明

## Dashboard 的閱讀邏輯

本章節說明：**業務主管在 Dashboard 中，應該如何閱讀畫面上的資訊，以及各區塊存在的意義。**

這裡不討論實作細節，而是先建立一個共同理解的閱讀順序。

---

### 一、Dashboard 不是報表，而是判斷工具

在本 CRM 中，Dashboard 的角色並不是彙整所有資料，而是協助業務主管完成三個動作：

1. 快速判斷目前整體狀況是否正常
2. 辨識哪些地方可能存在風險
3. 決定是否需要介入，以及從哪裡開始

因此，Dashboard 的設計重點不在於「資訊多寡」，而在於「閱讀效率」。

---

### 二、為什麼需要明確的閱讀順序

若沒有清楚的閱讀邏輯，Dashboard 常會出現以下問題：

- 各區塊資訊彼此獨立，無法形成判斷
- 使用者只挑自己熟悉的數字看
- 真正重要的警訊被大量資料淹沒

因此，本 Dashboard 採用「由整體到例外」的閱讀順序，而非隨機瀏覽。

---

### 三、整體閱讀順序說明

業務主管在打開 Dashboard 時，建議的閱讀流程如下：

1. **先看整體概況**  
   確認目前報價活動是否仍在正常流動

2. **再看異常與停滯**  
   找出哪些案件已超出合理進度

3. **最後看責任與介入點**  
   判斷是否需要分配資源或親自介入

這個順序，對應到 Dashboard 中由上而下的區塊配置。

---

### 四、為什麼不以個人或單筆案件為起點

對業務主管而言，若一開始就從「某位業務」或「某張報價」進入，容易導致兩種結果：

- 過早陷入細節，忽略整體風險
- 管理注意力被少數案件佔據

因此，本 Dashboard 刻意避免以「人」或「單筆案件」作為第一視角，而是先呈現整體狀態。

---

### 五、區塊之間的關係

Dashboard 中的各區塊並非獨立存在，而是層層遞進：

- 上層區塊負責「判斷是否正常」
- 中層區塊負責「指出問題在哪」
- 下層區塊負責「支援介入決策」

這樣的配置，能確保主管在最短時間內完成一次完整的管理判斷。

---

### 六、小結

Dashboard 的價值，不在於展示所有資訊，而在於**引導正確的閱讀順序**。

當閱讀順序被設計清楚，業務主管才能在有限時間內，快速理解狀況並做出判斷。
